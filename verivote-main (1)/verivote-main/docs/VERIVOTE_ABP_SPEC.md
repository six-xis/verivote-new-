# VeriVote-ABP Specification

VeriVote-ABP means Audit-Bound Partition Proof. The primary backend implementation is now `apps/api_py`. The Node backend in `apps/api` is legacy and is not the main target for new ABP features.

## Protocol Goal

ABP binds election identity, manifest, public audit roots, tally, and audit artifact hashes into the same proof public signals. The on-chain verifier path must check the submitted audit fields against the proof public signals before accepting the proof.

The target binding set is:

- `electionIdHash`
- `manifestHash`
- `commitmentRoot`
- `nullifierRoot`
- `receiptRoot`
- `tallyHash`
- `auditBundleHash`

## Phase 4 Eligibility And Nullifier Reference

Phase 4 adds Python reference helpers in `apps/api_py/app/crypto/eligibility.py` and
`apps/api_py/app/crypto/merkle.py`. This is not a ZK eligibility proof yet; it is the data
foundation that later circuits must prove.

### credentialSecret

`credentialSecret` is a high-entropy random string generated by the demo issuer. In Phase 4,
the backend stores it only for demo infrastructure and returns it once from
`/credentials/demo-issue`. It must not appear in bulletin-board responses, cast ballot
records, public audit bundles, or chain records.

In production, credentials should be issued by an independent eligibility authority and the
backend should not hold voter credential secrets.

### credentialCommitment

The Python reference commitment is:

```text
credentialCommitment = field_hash_v2(
  "VERIVOTE_CREDENTIAL_COMMITMENT_V1",
  [credential_secret]
)
```

This commitment is public and can be listed in the public credential set. It should not reveal
the underlying `credential_secret`.

### eligibilityRoot

`eligibilityRoot` is the Merkle root of the ordered public credential commitments for an
election. Empty credential sets use the fixed `EMPTY_ROOT`. Odd Merkle levels duplicate the
last node. The Python tree hashes leaves with `verivote.merkle.leaf.v1` and internal nodes with
`verivote.merkle.node.v1`.

Merkle proof items have:

| Field | Notes |
| --- | --- |
| `sibling` | Sibling node hash. |
| `position` | `left` when the sibling is left of the current node, otherwise `right`. |

The same ordered leaves produce the same root. Changing a credential commitment changes the
root. Future circuits must implement the same tree rules or explicitly migrate the rule hash.

### nullifierHash

The Python reference nullifier is:

```text
nullifierHash = field_hash_v2(
  "VERIVOTE_NULLIFIER_V1",
  [
    election_id_hash,
    credential_secret
  ]
)
```

The result is a BN254 field element encoded as a decimal string. It is election-scoped: the
same credential secret produces the same nullifier in the same election and a different
nullifier in a different election. `/ballots/cast` uses `nullifier_hash` to reject duplicate
casts within the same election.

This is a Python reference/demo hash. It must be replaced or aligned with the Poseidon-based
circuit implementation before production ZK claims are made.

### Future Private Valid Vote Proof

The future private valid vote proof should prove:

- the voter knows `credential_secret`;
- `credentialCommitment` is a member of `eligibilityRoot`;
- `nullifierHash = H(election_id_hash, credential_secret)`;
- `commitmentV2` binds the vote opening and `nullifier_hash`.

Phase 4 does not implement or claim this ZK proof.

## Phase 5 / M6A Private Valid Vote Proof Interface

M6A defines the backend proof interface and mock verifier guard for the future private valid
vote proof. It does not implement a real Circom/snarkjs verifier and must not be described as
a completed ZK privacy proof.

### PrivateValidVotePublicSignalsV1

| Field | Type | Notes |
| --- | --- | --- |
| `election_id_hash` | `str` | Current election binding. |
| `eligibility_root` | `str \| None` | Current election eligibility root. |
| `nullifier_hash` | `str` | Cast request nullifier. |
| `commitment` | `str` | Cast request commitment. |
| `rule_hash` | `str \| None` | Manifest rule hash, optional during migration. |

The fixed order is:

0. `election_id_hash`
1. `eligibility_root`
2. `nullifier_hash`
3. `commitment`
4. `rule_hash`

`public_signals` must not contain `vote_vector`, `randomness`, `candidate_id`, or
`credential_secret`.

### PrivateValidVoteProofV1

| Field | Type | Notes |
| --- | --- | --- |
| `proof` | `dict` | Proof object or mock proof object. |
| `public_signals` | `PrivateValidVotePublicSignalsV1` | Bound public signals. |
| `proof_system` | `str` | Current mock default is `mock-groth16`; real verifier later uses `groth16`. |
| `artifact_hash` | `str \| None` | Optional verifier/proving artifact hash. |
| `mock` | `bool` | Marks a mock proof. |

Mock proofs are accepted only when `app_mode` is `test` or `development` and
`zk_mock_mode=true`. Mock mode is forbidden in `competition` and `production`.

When `/ballots/cast` receives `validity_proof`, it checks:

- `public_signals.election_id_hash` equals the current election hash;
- `public_signals.eligibility_root` equals the current election eligibility root;
- `public_signals.nullifier_hash` equals the cast request nullifier;
- `public_signals.commitment` equals the cast request commitment;
- `public_signals.rule_hash` equals the manifest rule hash when provided;
- the ZK service accepts the proof.

The proofless cast path remains available only as a migration path. M6B/M7 should replace it
with mandatory real private valid vote verification.

## Phase 5 / M6C Private Valid Vote Circuit Pipeline

M6C runs the real Circom/snarkjs build, prove, and verify pipeline for the fixed
demo instance `private_valid_vote_4_8`, with four candidates and Merkle depth
eight. The Python snarkjs verifier wrapper can detect the generated
`verification_key.json` and verify real Groth16 proofs.

The circuit imports Poseidon with:

```circom
include "circomlib/circuits/poseidon.circom";
```

Build scripts pass `-l node_modules` so the include resolves through installed
`circomlib`.

### Public Inputs

The public signal order remains the M6A contract:

0. `election_id_hash`
1. `eligibility_root`
2. `nullifier_hash`
3. `commitment`
4. `rule_hash`

`vote_vector`, `randomness`, `candidate_id`, and `credential_secret` are not
public signals.

### Private Inputs

The private witness contains:

- `vote_vector[candidateCount]`
- `randomness`
- `credential_secret`
- `merkle_path_elements[depth]`
- `merkle_path_indices[depth]`

### Constraints

The circuit enforces:

- every vote vector entry is binary;
- the vote vector sum is exactly one;
- `credential_commitment = Poseidon([credential_secret])`;
- `nullifier_hash = Poseidon([election_id_hash, credential_secret])`;
- `eligibility_root` is recomputed from `credential_commitment` and the private
  Merkle path;
- `commitment` binds `election_id_hash`, `eligibility_root`,
  `nullifier_hash`, `rule_hash`, `vote_vector`, and `randomness`.

The commitment profile is layered:

```text
vote_hash = Poseidon(vote_vector...)
header_hash = Poseidon(election_id_hash, eligibility_root, nullifier_hash, rule_hash)
commitment = Poseidon(header_hash, vote_hash, randomness)
```

M6C uses `ZK_PROFILE = "poseidon-v1"`. The current Python `commitmentV2` and
`field_hash_v2` helpers are still SHA256-to-BN254 reference/demo helpers, so the
dynamic cast helper path is not yet fully aligned with this circuit profile.

M7 must force the cast API onto the real private valid vote proof path and align
cast commitment/nullifier generation with the Poseidon profile. This section
does not claim completion of batch tally proof or coercion-resistance coverage.

The current proof artifacts are development-only: the Powers of Tau and zkey are
generated locally by an unsafe setup and are not suitable for production or final
competition deployment.

### CastBallotRecordV2

| Field | Type | Notes |
| --- | --- | --- |
| `ballot_id` | `str` | Public ballot id. |
| `election_id_hash` | `str` | Proof binding id. |
| `commitment` | `str` | `commitmentV2`. |
| `nullifier_hash` | `str` | Duplicate prevention hash. |
| `sealed_vote_package` | `dict` | Public ciphertext/package metadata only. |
| `sealed_vote_package_hash` | `str \| None` | Hash binding the sealed package into later audit artifacts. |
| `validity_proof_hash` | `str \| None` | Optional proof digest. |
| `receipt_code` | `str` | Receipt code. |
| `receipt_chain_hash` | `str` | Receipt chain node hash. |
| `status` | `Literal["cast"]` | Fixed cast status. |
| `created_at` | `datetime` | Cast time. |

`CastBallotRecordV2` is a public cast record. It explicitly forbids plaintext vote fields:

- `candidate_id`
- `vote_vector`
- `randomness`

The Python model rejects these as extra top-level fields and also rejects them inside `sealed_vote_package`.

In the Phase 3 Python backend, the repository internally stores the full `sealed_vote_package`
inside `CastBallotRecordV2` so a later tally authority can decrypt it during tally/proof
generation. Public responses and the bulletin board do not return the full sealed package;
they return only this public projection:

| Field | Type | Notes |
| --- | --- | --- |
| `ballot_id` | `str` | Public cast id. |
| `election_id_hash` | `str` | Election binding. |
| `commitment` | `str` | `commitmentV2`. |
| `nullifier_hash` | `str` | Duplicate prevention hash. |
| `sealed_vote_package_hash` | `str` | Hash of the sealed package. |
| `validity_proof_hash` | `str \| None` | Placeholder proof digest until private valid vote proof is added. |
| `receipt_code` | `str` | Public receipt code. |
| `receipt_chain_hash` | `str` | Chain hash for the receipt record. |
| `status` | `Literal["cast"]` | Fixed cast status. |
| `created_at` | `str` | Cast time. |

`POST /api/v2/elections/{election_id}/ballots/cast` accepts only commitment data,
the sealed package, the sealed package hash, receipt code, nullifier hash, and the optional
proof hash placeholder. It rejects plaintext `candidate_id`, `vote_vector`, and `randomness`
at the request top level and inside `sealed_vote_package`.

The endpoint verifies:

- the election exists;
- `commitment`, `nullifier_hash`, `receipt_code`, and `sealed_vote_package_hash` are non-empty;
- `sealed_vote_package_hash == compute_sealed_vote_package_hash(sealed_vote_package)`;
- `nullifier_hash` has not already been used in the same election.

No fake ZK verifier is implemented in Phase 3. `validity_proof_hash` is only a boundary field
reserved for the future private valid vote proof.

### Phase 3 Receipt Chain

The current ABP v2 receipt chain uses a minimal reference hash:

```text
receipt_chain_hash = H(
  previous_receipt_chain_hash,
  election_id_hash,
  ballot_id,
  commitment,
  receipt_code,
  status
)
```

The first cast ballot uses `VERIVOTE_RECEIPT_GENESIS` as `previous_receipt_chain_hash`.
Changing `commitment`, `receipt_code`, or `status` changes the hash. M7 is expected to move
this into a fuller receipt-chain module and connect it to `receiptRoot`.

## Phase 2 Reference Hash and CommitmentV2

The Python backend now provides reference helpers in `apps/api_py/app/crypto`.

### canonical JSON

`canonical_json_dumps(data)` and `canonical_json_bytes(data)` produce stable JSON:

- sorted keys;
- no extra whitespace;
- datetime values serialized as ISO 8601 strings;
- Pydantic models serialized through `model_dump(mode="json")`;
- support for dict/list/str/int/bool/None.

### field_hash_v2

`field_hash_v2(domain, values)` is:

```text
field_hash_v2(domain, values)
  = SHA256(canonical_json({
      "version": "verivote-field-hash-v2",
      "hash": "sha256-to-bn254-reference",
      "domain": domain,
      "values": values
    })) mod BN254_FIELD_MODULUS
```

The output is a decimal string representing a valid BN254 field element.

This is a Python reference/demo hash. It is not a production Poseidon commitment. Future ZK circuits must replace it or align with a Poseidon implementation before production claims are made.

### commitmentV2

The current Python reference commitment is:

```text
commitmentV2 = field_hash_v2(
  "VERIVOTE_COMMITMENT_V2",
  [
    election_id_hash,
    nullifier_hash,
    *vote_vector,
    randomness
  ]
)
```

`vote_vector` must:

- have length equal to `candidate_count` when validated;
- contain only `0` or `1`;
- sum to exactly `1`.

Overvotes, undervotes, non-binary values, and length mismatch are rejected.

## SealedVotePackage

The Python reference sealed vote helpers live in `apps/api_py/app/crypto/sealed_vote.py`.

### VoteOpeningV2

| Field | Notes |
| --- | --- |
| `vote_vector` | Private one-hot vector. |
| `randomness` | Private commitment randomness. |
| `candidate_count` | Candidate count. |
| `election_id_hash` | Election binding. |
| `nullifier_hash` | Nullifier binding. |

### SealedVotePackageV1

Public sealed package:

| Field | Notes |
| --- | --- |
| `version` | Fixed `sealed-vote-v1`. |
| `algorithm` | Current value `AESGCM-SHA256-DEMO`. |
| `ciphertext` | Base64 AESGCM ciphertext. |
| `nonce` | Base64 AESGCM nonce. |
| `key_id` | Demo tally key id. |
| `opening_hash` | Hash binding the private opening. |
| `created_at` | Seal time. |

`compute_sealed_vote_package_hash(package)` hashes the canonical sealed package. This hash is intended to bind `sealed_vote_package` into `CastBallotRecordV2`, audit bundle artifacts, and later proof statements.

The demo AESGCM mode derives a symmetric key from `SHA256(demo_key)`. This is a demo tally encryption mode only; it is not the final production public-key tally encryption scheme. The `demo_key` must not appear in the sealed package, public audit bundle, or chain record.

The tally authority can decrypt sealed packages during tally/proof generation. Public observers and bulletin boards must not directly see `vote_vector`, `randomness`, or plaintext candidate choices.

The Phase 3 bulletin board returns `sealed_vote_package_hash`, not the full
`sealed_vote_package`; the full package remains repository-private until a tally authority
workflow consumes it.

## Current Baseline Boundary

`/api/v2/elections/{election_id}/ballots/legacy-cast` remains a migration endpoint. It preserves the current test baseline but is not the final private ABP cast API. `/ballots/cast` uses `CastBallotRecordV2` and must not persist plaintext vote fields outside the encrypted sealed package.
