# Private Valid Vote Proof Pipeline

Phase 5 / M6C runs the first real `private_valid_vote` Circom/snarkjs pipeline for the fixed `private_valid_vote_4_8` demo instance. This is a real Groth16 build/prove/verify path, but the setup and fixtures are still development/demo material.

- `circuits/private_valid_vote.circom`
- `circuits/private_valid_vote_4_8.circom`
- `scripts/zk/generate_private_valid_vote_input.mjs`
- `scripts/zk/build_private_valid_vote.ps1`
- `scripts/zk/prove_private_valid_vote.ps1`
- `scripts/zk/verify_private_valid_vote.ps1`
- `apps/api_py/app/zk/private_valid_vote.py`

## Include Path

The circuit imports Poseidon with:

```circom
include "circomlib/circuits/poseidon.circom";
```

Build scripts pass `-l node_modules` to `circom`, so the include resolves through the repository root `node_modules` directory. This is more portable than hard-coding a relative `../node_modules/...` path.

## Public Signals

The fixed public signal order is:

0. `election_id_hash`
1. `eligibility_root`
2. `nullifier_hash`
3. `commitment`
4. `rule_hash`

Public signals must not include:

- `vote_vector`
- `randomness`
- `candidate_id`
- `credential_secret`

## Private Inputs

The circuit private inputs are:

- `vote_vector`
- `randomness`
- `credential_secret`
- `merkle_path_elements`
- `merkle_path_indices`

## Poseidon Rules

The input generator uses `circomlibjs` Poseidon and must match the circuit:

```text
credential_commitment = Poseidon([credential_secret])
nullifier_hash = Poseidon([election_id_hash, credential_secret])
eligibility_root = Poseidon Merkle root for the private membership path
vote_hash = Poseidon(vote_vector...)
header_hash = Poseidon(election_id_hash, eligibility_root, nullifier_hash, rule_hash)
commitment = Poseidon(header_hash, vote_hash, randomness)
```

The generated fixture uses a fixed depth-8 Merkle tree with 256 leaves and deterministic non-member leaves. It writes:

- `circuits/inputs/private_valid_vote.valid.json`
- `circuits/inputs/private_valid_vote.invalid_overvote.json`
- `circuits/inputs/private_valid_vote.invalid_membership.json`
- `artifacts/zk/private_valid_vote/public_signals_expected.json`

## Windows Commands

```powershell
powershell -ExecutionPolicy Bypass -File scripts/zk/build_private_valid_vote.ps1
powershell -ExecutionPolicy Bypass -File scripts/zk/prove_private_valid_vote.ps1
powershell -ExecutionPolicy Bypass -File scripts/zk/verify_private_valid_vote.ps1
```

`verify_private_valid_vote.ps1` runs:

```text
snarkjs groth16 verify verification_key.json public.json proof.json
```

The scripts quote paths correctly for Windows workspaces containing spaces or parentheses.

## Python Wrapper

The Python wrapper reports the real verifier as available when:

- `verification_key.json` exists;
- `snarkjs` is callable;
- mock mode is disabled.

It verifies real proofs through `snarkjs groth16 verify`. It never falls back to mock in `competition` or `production`.

## Still Unsafe

- The local Powers of Tau and zkey are generated by an unsafe development setup.
- The fixture values are deterministic demo values.
- Python reference hash helpers still exist and are not the same as the Poseidon circuit profile.
- Batch tally proof is not implemented.
- M7 should wire real proof generation/verification into the dynamic cast API path and remove proofless migration acceptance.
